> - 生成ルール: **12〜16ターン**の会話を生成

**実装の現状:**


### A-3. 印象（feeling label）の系列が仕様の2系列制ではなく単一ラダーになっている【乖離】

**仕様書:** `03_関係・感情系.md:41-76`
> 印象は**非対称・一方向**。関係タイプに応じて**2系列**を使い分ける。

**仕様の2系列:**

**通常系列**（`relation` が `none` / `acquaintance` / `friend` のとき使用）:
```
嫌いかも(maybe_dislike) ← なし(none, 初期値) → 気になる(curious) → 好きかも(maybe_like)
```
4段階。初期値は `none`。

**特別系列**（`relation` が `best_friend` / `lover` / `family` のとき使用）:
```
嫌い(dislike) ← 普通(none) ← 好き(like, 初期値) → 大好き(love)
```
4段階。初期値は `like`。

**系列切替ルール** (`03:71-75`):
- **昇格時**（friend → best_friend/lover）: 例外なく双方の印象を `like` にリセット
- **降格時**（best_friend/lover → friend）: 好感度に基づく自動付与テーブルで通常系列の値を決定

**実装の現状** (`apps/web/lib/evaluation/evaluate-conversation.ts:75`):
```ts
const IMPRESSION_ORDER: Impression[] = [
  'dislike', 'maybe_dislike', 'none', 'curious', 'maybe_like', 'like'
];
```

| 項目 | 仕様 | 実装 |
|------|------|------|
| 系列数 | 2系列（通常系列 / 特別系列）を関係タイプで切替 | 全関係タイプで共通の**単一ラダー**（6段階） |
| `love` レベル | 特別系列の最上位。`like` の上に位置する | DB の `feeling_label` enum には `'love'` が定義されているが、評価エンジンの `IMPRESSION_ORDER` に含まれておらず、**到達する経路が存在しない** |
| 通常系列で使わない値 | `dislike` と `like` は特別系列専用 | 単一ラダーに全値が混在しており、`friend` 関係でも `dislike` や `like` に到達しうる |
| 昇格時リセット | friend → best_friend/lover 昇格時に双方を `like` にリセット | リセット処理なし。昇格前の印象がそのまま引き継がれる |
| 降格時変換 | best_friend/lover → friend 降格時に好感度ベースで通常系列の値を自動付与 | 変換処理なし |

**何が問題か:** 仕様では、通常の友達関係と親密な関係で使う印象ラベルを分けることで、「友達同士の好きかも」と「恋人同士の大好き」のニュアンスの違いを表現する設計になっている。単一ラダーではこの区別がつかず、関係の昇格・降格時の印象リセットも機能しない。

**修正が必要な箇所:**
1. `apps/web/lib/evaluation/evaluate-conversation.ts` — 関係タイプに応じて2系列のラダーを切り替えるロジック
2. `apps/web/lib/evaluation/weights.ts` — `ImpressionBase` 型に `'love'` を追加
3. 関係変更処理（管理室の手動変更 + 将来の自動遷移）に印象リセット/変換ロジックを追加

### A-4. 印象の変動判定が「直近3件の重み付き窓」ではなく「毎回の単発判定」になっている【乖離】

**仕様書:** `03_関係・感情系.md:79-107`
> 直近3件の会話のfavorデルタを重み付き合計し、閾値を超えたら1段階変化。

**仕様の評価ロジック:**
```
impressionScore = delta1 × 4 + delta2 × 3 + delta3 × 2
```
- `delta1` = 直近の会話のfavorデルタ、`delta2` = 1つ前、`delta3` = 2つ前
- 重み `[4, 3, 2]`（直近が44%、1つ前が33%、2つ前が22%）
- score理論範囲: `-18` 〜 `+18`（favorデルタが `[-2, +2]` にクリップされるため）
- **変化閾値**: `+7` で1段階UP、`-7` で1段階DOWN
- **3件蓄積するまで印象は変化しない**
- **変化が起きたら追跡窓をリセット**（0件から再蓄積開始）

**実装の現状** (`apps/web/lib/evaluation/evaluate-conversation.ts:150-154`):
```ts
const deltaSignA2B = a2bFavor >= 1 ? 1 : a2bFavor <= -1 ? -1 : 0;
const deltaSignB2A = b2aFavor >= 1 ? 1 : b2aFavor <= -1 ? -1 : 0;
let nextBaseA2B = nextImpression1step(prevStateA2B.base, deltaSignA2B);
let nextBaseB2A = nextImpression1step(prevStateB2A.base, deltaSignB2A);
```

**差分:**
| 項目 | 仕様 | 実装 |
|------|------|------|
| 評価対象 | 直近3件の会話のfavorデルタの重み付き合計 | 今回の会話のfavorデルタのみ |
| 変化閾値 | `±7`（蓄積スコアに対して） | `±1`（今回のfavorデルタに対して） |
| 蓄積要件 | 3件蓄積してから初めて評価 | 蓄積なし。毎回判定 |
| 窓リセット | 変化後にリセット | リセット不要（窓が存在しない） |

**何が問題か:** 仕様では印象は「じわじわと変化する」設計（3回分の会話の傾向を見て判定）だが、実装では1回の会話で即座に変化する。これにより印象変動が激しくなりすぎ、「日常系」の穏やかな変化という設計意図から外れる。

**修正に必要な追加要素:**
1. 直近3件のfavorデルタを保持する仕組み — `feelings` テーブルにJSONBカラム（例: `recent_deltas`）を追加するか、`events` テーブルの直近3件の会話イベントから `deltas.aToB.favor` / `deltas.bToA.favor` を取得して算出
2. 重み付き合計と閾値 `±7` での判定ロジック
3. 3件未満のときは印象変化をスキップする処理
4. 変化発生後の窓リセット処理

### A-5. 好感度デルタの「3層乗算補正」（感度×スタンス×話題）が未実装【未実装】

**仕様書:** `03_関係・感情系.md:188-253`
> `A→Bのdelta = Math.round(baseFavor × sensitivityA × stanceModifierA × topicBonus)`
> 結果は `[-2, +2]` にクリップ。

仕様では、会話評価エンジンが算出した `baseFavor`（タグ重み等の加算結果）に対して、以下の3つの**乗算**補正を適用し、**キャラごとに異なるデルタ**を生み出す:

**① 感度係数（sensitivity）** (`03:203-233`) — キャラ固有の受け取りやすさ。作成時に確定し変動しない:
```
sensitivity = 1.0
  + (empathy - 3) × 0.10       // 共感力が高いほど影響を受けやすい
  + (expressiveness - 3) × 0.05 // 表現力が高いほど影響を受けやすい
  + (stubbornness - 3) × -0.05  // 頑固さが高いほど影響を受けにくい
  + mbtiBonus                    // F型: +0.10, T型: -0.10
```
理論範囲: `0.50〜1.50`、実質帯: `0.7〜1.3`

**② スタンス補正（stanceModifier）** (`03:235-245`) — 今回の会話でのスタンスに応じた係数:
| スタンス | 補正 | 意味 |
|---|---|---|
| enthusiastic | ×1.3 | 乗り気なので影響大 |
| agreeable | ×1.0 | 標準 |
| reluctant | ×0.7 | 乗り気でないので影響小 |
| indifferent | ×0.5 | 無関心なので影響半減 |
| confrontational | ×1.2 | 対立的だが感情は動く |

**③ 話題ソース補正（topicBonus）** (`03:247-253`) — 話題の性質による係数:
| 話題ソース | 該当キャラ | 補正 | 意味 |
|---|---|---|---|
| self_experience | 体験を話す側 | ×1.2 | 自分事なので影響大 |
| heart_to_heart | 両方 | ×1.1 | 内面の話は双方に影響 |
| その他 | — | ×1.0 | 標準 |

**実装の現状** (`apps/web/lib/evaluation/evaluate-conversation.ts:106-132`):

現在の好感度デルタ算出は以下の**加算のみ**のパイプライン:
1. タグ重みの合計（共感+0.6, 感謝+0.7, 称賛+0.8 等）
2. 会話バランスボーナス（最大+0.2）
3. 品質ヒント加点（coherence.good +0.2, tone.gentle +0.2 等）
4. `[-2, +2]` にクリップ

→ **A→BとB→Aで同じデルタ値**になる（非対称にならない）。sensitivity / stanceModifier / topicBonus の乗算は一切行われていない。

**何が問題か:** 仕様の核心的な設計意図は「同じ会話でもキャラの性格やスタンスによって受け取り方が異なる」という**非対称化**にある。現在の実装ではA→BとB→Aのデルタが常に同じであり、キャラクターの個性が好感度変動に反映されない。

**修正が必要な箇所:**
1. `apps/web/lib/evaluation/evaluate-conversation.ts` — `evaluateConversation` 関数にキャラプロフィール（性格パラメータ・MBTI）と `ConversationStructure`（スタンス情報）と話題ソースを入力として追加
2. sensitivity 算出関数の新規実装
3. baseFavor算出後に `Math.round(baseFavor × sensitivity × stanceModifier × topicBonus)` を適用し、`[-2, +2]` にクリップ
4. A→BとB→Aで独立に算出

### A-6. 好感度の時間経過による自然減少（中立点30への収束）が未実装【未実装】

**仕様書:** `03_関係・感情系.md:267-276`
> 好感度は会話がなくても時間経過で**中立点30に向かって収束**する。

| 最終接触からの経過日数 | 好感度 > 30 のとき | 好感度 < 30 のとき |
|---|---|---|
| 3日以内 | 変動なし | 変動なし |
| 4〜7日 | **-1/日** | **+1/日** |
| 8日以上 | **-2/日** | **+2/日** |

好感度 = 30 の場合は変動なし。整数単位で変動し、30を超えて逆方向にはならない。

**実装の現状:** 好感度（`feelings.score`）を時間経過で変動させるコードは存在しない。好感度は `evaluateConversation` 関数（会話評価時）でのみ変動する。

**何が問題か:** 会話が行われないペアの好感度が永久に固定される。仕様では「放置すると関係が薄まる」という自然な関係変化を表現する設計だが、現在は一度上がった好感度が下がらない。

**修正に必要な追加要素:**
1. 最終接触日時の取得方法 — `events` テーブルから該当ペアの直近の `kind='conversation'` イベントの `updated_at` を取得するか、`feelings` テーブルに `last_contact_at` カラムを追加
2. 日次の定期実行バッチ — クライアントサイドスケジューラー（会話・天気スケジューラーと同様の仕組み）またはサーバーサイドcron
3. 中立点30への収束ロジック — 現在のscoreと30の差分方向に、経過日数に応じた速度で変動

### A-7. 約束（promise）システムが未実装、旧来のsignals/pausedが残存【未実装 + 残存】

**仕様書:** `04_会話生成系.md:238-303`

> 旧仕様の `paused` およびGPTの `signals` 出力は**廃止** (`04:249`)。

**仕様の約束生成システム** (`04:252-289`):

会話パイプラインの話題選定後・GPT呼び出し前に約束フラグを抽選:
```
P(約束) = 10% × 関係重み × 話題重み × favorModifier
```

関係重み (`04:261-268`):
| 関係 | 重み |
|---|---|
| none / acquaintance | 0（約束なし） |
| friend | 1.0 |
| family | 1.1 |
| best_friend / lover | 1.3 |

話題重み (`04:270-280`):
| 話題ソース | 重み |
|---|---|
| shared_interest | 1.1 |
| personal_interest | 1.0 |
| snippet | 0.8 |
| seasonal | 0.7 |
| self_experience | 0.5 |
| small_talk | 0.4 |
| continuation / third_party / heart_to_heart | 0（約束なし） |

好感度補正: `favorModifier = 0.5 + (favor / 100)`（範囲 0.5〜1.5）
実効レンジ: 約2〜17%

**3種の会話タイプ** (`04:291-301`):
| タイプ | 約束フラグ | 話題ソース | threadStatus | プロンプトへの指示 |
|---|---|---|---|---|
| 通常会話 | OFF | continuation以外 | `done` | 「約束を作らないこと」 |

**実装の現状:**

3種の会話タイプの分岐: **存在しない**。全会話が同じプロンプトで生成される。

- **データ構造:** 仕様書 Section 6.3 に詳細記載済み

**実装の現状:**
