# 06. プレイヤー介入・インフラ系

---

## 6.1 プレイヤーの立ち位置

- **管理人**として世界を見守る立場。キャラクターにとっては創造主・上位存在
- キャラクターとの接触が可能（相談など）
- キャラクターの認識は**グラデーション**: 信頼できる相談相手 〜 なんか見られているらしい程度
- 信頼度パラメータがこの認識のグラデーションを表現する

---

## 6.2 通知システム

ゲーム内で発生した会話・相談・システムイベントをプレイヤーに知らせるインフラ。

### 通知の種類

| type | 説明 |
|---|---|
| conversation | 会話イベント発生 |
| consult | 住人からの相談 |
| system | システム通知 |

### データ構造

```
notifications テーブル:
  id (UUID PK), type (text),
  linkedEventId (UUID), threadId (UUID nullable),
  participants (JSONB nullable),
  snippet (text, 最大60文字 + '…'),
  occurredAt (timestamp), status (text: unread/read/archived),
  priority (integer, default 0),
  updatedAt (timestamp), ownerId (UUID)
```

### 通知のライフサイクル

1. **生成**: 会話/相談イベント作成時に自動生成（`snippet` = 会話の1行目を60文字に切り出し）
2. **表示**: IndexedDBから即座に表示 → 非同期でクラウド取得 → マージ
3. **既読化**: クリック時に `unread → read`（ローカル + リモート）
4. **フィルタリング**: `read` は5時間以内、`unread` は10時間以内のもののみ表示
5. **ページネーション**: 5件/ページ

### クリック時の動作

- `consult` → 相談詳細パネルを開く
- `conversation` → 会話ログを開く

遷移先は `linkedEventId` を使用して該当イベントに直接遷移する（F-6: 現行の未活用を解消）。

**実装参考**: `notifications` テーブル、`apps/web/lib/data/notifications.ts`、`apps/web/components/notifications/NotificationsPanel.tsx`。

## 6.3 相談システム（consult）

住人がプレイヤーに対して「相談」を持ちかけ、プレイヤーが選択肢から回答することで信頼度が変動する、唯一のプレイヤー↔住人間のインタラクション。

### 2種類の相談

#### 日常相談

| 項目 | 仕様 |
|---|---|
| 発生条件 | 信頼度に応じた頻度と深さ |
| 信頼度が低い場合 | そもそも相談が来ない |
| 深さ | 「ちょっと深いけど重すぎない程度」 |
| 結果 | 信頼度の変動（主な変動経路） |

#### 関係遷移相談（告白・別れ）

| 項目 | 仕様 |
|---|---|
| 発生条件 | 好感度・印象ベース（信頼度は発生条件に含めない） |
| 信頼度の影響 | 発生条件には含めないが、**相談のトーン・態度に反映** |
| 信頼度高 | 素直に相談してくる |
| 信頼度低 | ぎこちなく、「別に相談とかじゃないんだけど…」的な態度 |
| 結果 | 関係遷移 + 信頼度変動 |

### 信頼度の段階に応じた態度変化

前身Specの4段階: **警戒** → **距離あり** → **好意的** → **親密**

| 信頼度帯 | トーン |
|----------|--------|
| 0-24 | 警戒的（相談対象外） |
| 25-49 | よそよそしい |
| 50-74 | 好意的 |
| 75-100 | 打ち解けた・親密 |

### 生成方式（設計決定事項）

**2段階生成方式を採用:**

1. **相談テキスト（お題）生成**: タブを開いた時点でGPTコールにより生成
2. **返答テキスト生成**: プレイヤーが回答した後にGPTコールで生成

理由: 将来的にプレイヤーの自由記述回答への対応を見据え、返答の事前生成は行わない。

### 相談テーマ生成 — カテゴリ＋シード方式

GPTへの丸投げではなく、コード側で方向性を制御してからGPTに肉付けさせる。

1. **コード側でカテゴリを重み付き抽選**（例: 人間関係 / 日常の悩み / 将来への不安 / 雑談・軽い話題）
2. **カテゴリ内でシードを抽選**（例: 人間関係 → 「特定の相手との距離感」「グループ内での居場所」「すれ違い」）
3. **GPTに「カテゴリ＋シード＋キャラ情報＋信頼度帯トーン」を渡して肉付け**

信頼度帯によるカテゴリ出現率:

| 信頼度帯 | 重いテーマ | 軽いテーマ |
|----------|-----------|-----------|
| 25-49 | 低い | 高い |
| 50-74 | 中程度 | 中程度 |
| 75-100 | 高い | 低い |

### 発生フロー

```
タブ開く
  → 即座に発生判定（75%の確率）
  → 未処理の relation_trigger イベントあり？
      YES → そのキャラで関係遷移相談を生成（最優先）
      NO  → 対象キャラを信頼度＋性格で重み付き抽選
  → GPTで相談テキスト生成（カテゴリ＋シード方式）
  → 約1分後に通知表示
  → 以降30-60分おきに同じ判定を繰り返し
```

### 頻度制御

| ルール | 値 |
|--------|-----|
| 発生確率 | 75%（チェックごと） |
| キャラ単位の制限 | 1人1日1回まで |
| チェック間隔 | 初回は即座、以降30-60分おき |
| 未回答の有効期限 | 10時間（経過で自動消滅、回答不可） |

### 抽選の重み付け

```
weight = trustBandWeight × personalityWeight
```

信頼度帯の重み:

| 信頼度帯 | trustBandWeight |
|----------|----------------|
| 0-24 | 0（対象外） |
| 25-49 | 1 |
| 50-74 | 3 |
| 75-100 | 5 |

性格補正:

```
personalityWeight = 1.0
  + (sociability - 3) × 0.15
  + (empathy - 3) × 0.1
  + (stubbornness - 3) × -0.1
```

### 信頼度変動（trustDelta）

GPTには返答テキストと回答の**好ましさ**（positive / neutral / negative）のみを判定させ、trustDelta はコード側で算出。

```
trustDelta = round(baseDelta × modifier)
```

基礎値:

| 好ましさ | baseDelta |
|----------|-----------|
| positive | +3 |
| neutral | +0.5 |
| negative | -3 |

性格補正:

```
modifier = 1.0
  + (empathy - 3) × 0.1
  + (stubbornness - 3) × -0.1
  + (expressiveness - 3) × 0.05
```

結果レンジ: 動きにくいキャラで ±2、平均で ±3、感受性高いキャラで ±4 程度。

### 関係遷移相談の選択肢設計

#### 告白相談

| 選択肢 | 好ましさ | 結果 |
|--------|----------|------|
| 応援A（積極的） | positive | 告白実行、成功率にボーナス |
| 応援B（控えめ） | neutral | 告白実行、成功率はベースのまま |
| 止める | negative | 告白不発生 |

#### 別れ相談

| 選択肢 | 好ましさ | 結果 |
|--------|----------|------|
| 賛同A（背中を押す） | positive | 別れ成立 |
| 賛同B（消極的同意） | neutral | 別れ成立 |
| 止める | negative | 別れ不成立 |

#### 「止める」場合の再発火防止

- **印象リセット**: 告白を止めた場合 `maybe_like → curious` に降格、別れを止めた場合 `dislike → none` に降格
- **クールダウン**: 同ペア・同トリガー種別で7日間は再発行しない
- 両方を併用し、自然な頻度に抑える

### 相談イベントのデータ構造

**生成時:**

```
events テーブル (kind: 'consult'):
  payload: {
    title: string,
    content: string,
    choices: [{ id, label }],
    residentId: uuid,
    participants: uuid[],       // 関係遷移相談の場合の関係者
    triggerEventId?: string,    // relation_trigger との紐付け
    category: string,           // テーマカテゴリ
    seed: string,               // テーマシード
    trustBand: string,          // 生成時の信頼度帯
    expiresAt: string           // 有効期限（生成から10時間後）
  }
```

**回答後（同イベントを更新）:**

```
  payload: {
    ...上記,
    selectedChoiceId: string,
    reply: string,              // GPT生成の返答テキスト
    favorability: 'positive' | 'neutral' | 'negative',  // GPT判定
    trustDelta: number,         // コード側算出
    trustBefore: number,
    trustAfter: number,
    answeredAt: string
  }
```

### 回答のUIフロー

1. 選択肢をクリック → 一時選択（`picked`、何度でも変更可能）
2. 「回答する」ボタン → 確定（`committed`、不可逆）
3. 確定後: GPTで返答テキスト生成 → 住人の返答テキスト表示 + 信頼度変化結果表示

### 二重回答防止

既に回答済みの相談に再度回答はできない。UIも確定後は全選択肢を `disabled` 化。

## 6.4 覗く機能（SectionBの決定事項）

プレイヤーが任意の住人の「今の様子」を能動的に見られる機能。会話や相談をすべて見終えた後の暇な時に触れる要素。「今何してるのかな？」という愛着に繋がる。

| 項目 | 仕様 |
|---|---|
| 生成方法 | LLMで生成（軽量モデル使用） |
| 入力 | 住人の性格・趣味・職業・時間帯・最近の出来事など |
| 住人の認識 | 覗かれていることを認識しない（「たまに覗かれているらしい」程度） |
| ゲームへの影響 | なし（見るだけ、データ更新なし、イベント記録なし） |

### 生成内容（設計決定事項）

2つの枠を別々に生成する（UIに状況描写用と独り言用の枠が既に用意されている）:

| 枠 | 内容 | 例 |
|----|------|-----|
| 状況描写 | 今何をしているかの短文 | 「リビングでコーヒーを飲みながら、窓の外を眺めている。」 |
| 独り言 | キャラらしい一言 | 「…明日の天気、どうだったっけ。」 |

心情描写は含めない。

### GPTへの入力（リッチ構成）

- 性格パラメータ、MBTI
- 趣味、職業
- 時間帯、天気
- 最近の会話の出来事、感情状態、関係状況

プロンプトで「直近の出来事は参考程度に。日常の一コマを描写すること」と指示し、最近の会話に引っ張られすぎないようにする。コンテキスト量はあとから調整可能。

### キャッシュ

- 覗くボタン押下 → GPTコール → 結果を返却
- 同じキャラは**30分間キャッシュ**（30分以内の再覗きは同じ内容を表示）
- キャッシュはクライアント側（メモリ or localStorage）

**実装参考**: `apps/web/components/room/home-content.tsx`（`HomePanelMode: 'right-peek'` のUIスタブ）。

## 6.5 手動設定（管理室 Office）

プレイヤーが住人の属性やペア間の関係性を手動で作成・編集するための管理画面。

### 管理できる項目

| カテゴリ | 項目 |
|---|---|
| 住人基本情報 | 名前, MBTI, 性格パラメータ, 性別, 年齢, 誕生日, 趣味 |
| 表現スタイル | 話し方プリセット, 一人称, 職業 |
| 生活リズム | 就寝時刻, 起床時刻 |
| 関係設定 | 関係ラベル, 印象（双方向）, 好感度（双方向）, 呼び方（双方向） |

### プリセットの find-or-create

ユーザーはプリセットの「ラベル名」を入力/選択し、保存時に find-or-create でUUIDに変換。見つからなければ新規プリセットを自動作成。

### 関係データの一括保存

各ペアに対して5つのテーブル（relations / feelings×2方向 / nicknames×2方向）を並列更新。

### 住人削除

ソフトデリート（`deleted: true`）。天気コメントの担当者だった場合はコメントを `null` にクリア。

**実装参考**: `apps/web/app/(dashboard)/office/` 配下のページ群。

## 6.6 データ同期システム

IndexedDB（クライアント）⇔ Supabase（サーバー）間の双方向データ同期。

### 設計方針

- **ローカルファースト**: 全操作はまずIndexedDBに書き込み → 非同期でサーバー同期
- **LWW（Last Writer Wins）**: `updated_at` タイムスタンプが新しい方が勝つ競合解決
- **Outboxパターン**: ローカル変更をキューイングし、オンライン時に送信
- **ソフトデリート**: `deleted` フラグで論理削除
- **`owner_id` ベースのRLS**: サーバー側で `owner_id = user.id` を強制上書きし、ユーザー分離

### 同期対象テーブル（8テーブル）

`presets`, `residents`, `relations`, `feelings`, `nicknames`, `events`, `consult_answers`, `world_states`

### 同期トリガー（5種類）

1. **online/offline イベント**: ネットワーク復帰時
2. **Supabase Realtime**: 8テーブルの `postgres_changes` を購読
3. **Auth状態変化**: サインイン/サインアウト/トークンリフレッシュ
4. **明示的リクエスト**: `window.dispatchEvent('reladen:request-sync')`
5. **手動ボタン**: UIの「↻ 同期」ボタン

### エラーリカバリ

- 指数バックオフ: `base × 2^retryCount + jitter` で最大60秒待ち
- 最大5回リトライ → 超過でエラーフェーズに遷移
- 認証エラー: エラー表示せずリトライ

### セキュリティ

- **owner_id 強制注入**: `owner_id` を持つテーブルのみ、サーバー側で `owner_id = user.id` を上書き
- **許可カラムリスト**: テーブルごとに pushで許可されるカラムを制限
- **consult_answers**: クライアント ⇔ サーバーの push/pull 対象。pushは `insert` ベースの first-write-wins（同一 `id` 再送は no-op）
- **consult_answers カラム互換**: `selected_choice_id/decided_at` と `selectedchoiceid/decidedat` の両系統を受理
- **camelCase→snake_case 自動変換**: クライアント⇔サーバー間のギャップを吸収

### 3つのストレージ層

| 層 | 用途 |
|---|---|
| IndexedDB（メイン） | `reladen` DB — 全テーブルのデータ |
| Outbox IndexedDB | `reladen-sync` DB — 送信キュー |
| Supabase | クラウド永続化 |

### Tauri対応

Tauri環境では IndexedDB の代わりにファイルシステムベースのJSONストレージに透過的に切り替え。デスクトップアプリ展開が可能。

**実装参考**: `apps/web/app/api/sync/[table]/route.ts`、`apps/web/lib/sync/outbox.ts`。
