# 04. 会話生成系

---

## 4.1 会話自動スケジューリング

### 現行仕様（クライアントサイド）

クライアントサイドで動作するタイマーベースのスケジューラー。15分間隔（±20%ジッター）で会話ペアを選定し、サーバーAPIに会話生成をリクエストする。

- マルチタブの重複実行を `localStorage` ロックで抑止
- 睡眠中の住人を除外した上で、ongoing状態のスレッド（古い順）を優先してペア選定
- `relation = none` のペアはスキップ
- 3段階フォールバック: ①既存スレッド → ②同ペア新規 → ③別ペア新規
- 認証切れ（401）時に自動停止

### 移行予定

> **サーバーサイドでの1時間間隔自動生成に移行予定**（リリースまでに対応。現行のクライアントサイド15分間隔は開発用）。

**実装参考**: `apps/web/lib/scheduler/conversation-scheduler.ts`（スケジューラー）、`apps/web/lib/conversation/candidates.ts`（候補選定）。

## 4.2 会話生成パイプライン

会話生成の全工程を統括するオーケストレータ。APIエントリポイント（I/O依存）とコアロジック（純粋ロジック）の二層構成。

### 11段階パイプライン

```
① 参加者解決（threadId or participants）
② プロフィール読み込み（speechPreset → キャッシュ or LLM抽出）
③ 関係性読み込み（type='none' なら中止）
④ 好感度読み込み（双方向の label + score）
⑤ 環境決定（シチュエーション生成）
⑥ 会話履歴読み込み（前回の記憶 + 直近5件の話題）
⑦ バッチ生成（スニペット鮮度チェック + 直近イベント鮮度チェック）
⑧ コンテキスト取得（24h以内のスニペット + 各キャラの知識）
⑨ 話題選定 + 約束フラグ抽選（§4.7）+ 会話構造決定 + GPT呼び出し
⑩ 評価 + スレッドステータス決定（約束フラグに基づく）
⑪ 永続化
```

### シチュエーション生成（SectionBの決定事項）

現行の固定マッピング（朝=カフェ, 昼=商店街, 夕=公園, 夜=コンビニ）+ 5パターンの出会いシチュエーションを**廃止**し、GPT-4o-miniで毎回異なるシチュエーション描写を生成する。

| 項目 | 仕様 |
|---|---|
| LLMモデル | GPT-4o-mini |
| 入力 | キャラ情報（名前・職業・趣味）、関係ラベル、時間帯、日付、直近シチュエーション数件 |
| 出力 | 20〜30文字の状況描写 |
| 重複回避 | 直近数件との重複回避あり |

**実装参考**: `apps/web/lib/conversation/run-conversation.ts`（パイプライン）、`apps/web/app/api/conversations/start/route.ts`（APIエンドポイント）。

## 4.3 話題選定エンジン

9種のソースから話題候補を生成し、スコアを重みとした**重み付きランダム選択**で1話題を選定する。

### 選定方式

- 全候補のスコアを重みとした重み付きランダム選択（「同率最高スコア群からランダム選択」から変更）
- スコア0以下の候補は除外

### 9種の話題ソース

| ソース | 内容 | 元データ | 親密度との関係 |
|---|---|---|---|
| shared_interest | 共通の趣味 | interests積集合 | 中程度 |
| personal_interest | 片方の趣味 | interests差集合 | 中程度 |
| continuation | 前回の続き | unresolvedThreads | 問わない |
| snippet | 共有体験 | shared_snippets | 中程度 |
| third_party | 第三者の噂 | offscreen_knowledge | 中程度 |
| self_experience | 自分の最近の出来事 | recent_events | 中程度 |
| 自己開示・質問 | 内面の話、相手を知る | カテゴリのみ | **高親密度で強い** |
| 世間話 | 軽い日常会話 | カテゴリのみ | 低くてもOK |
| 季節・時事 | 季節の話題 | カテゴリのみ | 低くてもOK |

> ※ 旧仕様の `feeling_shift` は削除。`environmental` は「世間話」に置換。
> ※ 世間話・季節・時事・自己開示は「カテゴリのみ選定し、具体的な内容はGPTに委ねる」方式。

### スコアリング

各候補に基本スコア + 親密度補正 + 特性補正 + 低親密度ペナルティ + 鮮度ペナルティ（-3）を適用。`continuation` の基本スコアは7（最高）で維持。重み付きランダムにより一強にはならない。

### 主導者ベースの候補生成

- `personal_interest` は initiator の興味のみから生成
- `third_party` は initiator の知識のみから生成
- 追随者の固有情報は候補に入らない

**実装参考**: `packages/shared/logic/topic-selection.ts`。

## 4.4 会話構造決定

「誰が話を振るか」「それぞれどんな態度か」「会話全体の雰囲気はどうか」「発話の長さはどの程度か」を**コードで決定論的に**算出する。

### 4段階の決定プロセス

**① 主導者決定**

- `personal_interest` → その興味を持っている側が主導
- その他 → ドミナンススコアで判定:
  - `sociability×0.35 + activity×0.3 + expressiveness×0.25 + stubbornness×0.1`
  - 差0.5未満 → ID辞書順（テスト容易性）、差0.5以上 → 高い方が主導

**② 感情スタンス決定**

5種のスタンス: **enthusiastic**（乗り気）/ **agreeable**（穏やか）/ **reluctant**（面倒）/ **indifferent**（興味薄）/ **confrontational**（突っかかり）

性格パラメータ + 好感度 + 話題興味 + 主導者フラグから5種への傾きスコアを算出し、最大値を採用。

**②-a. 話題関心度によるfavor影響調整**

話題ソース × キャラの立場で「関心度」を判定し、関心が低い場合はfavorのスタンスへの寄与を減衰させる（例: ×0.5）。好感度が高くても興味が薄い話題なら盛り上がりにくい、という自然なばらつきを生む。

**関心が高い**（favorがそのまま効く）:

- `shared_interest`（共通の趣味）
- `personal_interest`（興味を持っている側）
- `continuation`（前回の続き）
- `self_experience`（自分の体験を話す側 / 聞く側どちらも）
- `自己開示・質問`

**関心が低い**（favorの影響を減衰）:

- `personal_interest`（聞かされる側）
- `third_party`（第三者の噂）
- `世間話`
- `季節・時事`

> ※ 減衰の具体的な係数（×0.5等）はテストプレイで調整

**②-b. 日常ノイズ（気分のばらつき）**

スタンス確定後、各キャラ**独立に25%の確率**でスタンスが1段階ニュートラル方向にシフトする。日によるコンディションの差を表現し、同じ条件でも毎回同じ結果にならない揺らぎを与える。

| シフト前 | シフト後 |
|---|---|
| enthusiastic | agreeable |
| agreeable | indifferent |
| confrontational | reluctant |
| reluctant | indifferent |
| indifferent | 変化なし |

| ケース | 確率 |
|---|---|
| 両方通常 | 56% |
| 片方シフト | 38% |
| 両方シフト | 6% |

**③ 温度感決定**

4種の温度感: **warm** / **lukewarm** / **neutral** / **tense**

| 条件 | 温度感 |
|---|---|
| confrontational あり | tense |
| 両者 positive + 親しい関係 | warm |
| reluctant/indifferent あり | lukewarm |
| それ以外 | neutral |

**④ 発話長決定**

| スタンス | 主導者 | 追随者 |
|---|---|---|
| enthusiastic | 1〜2文 | 1〜2文 |
| agreeable | 1〜2文 | 1文 |
| reluctant | 1文 | 1文以内 |
| indifferent | 1文以内 | 1文以内 |
| confrontational | 1〜2文 | 1〜2文 |

**実装参考**: `packages/shared/logic/conversation-structure.ts`。

## 4.5 一発話の長さ制御（SectionBの決定事項）

| 項目 | 値 |
|---|---|
| 目安 | 20〜30文字 |
| 上限 | 40文字 |
| 制御方法 | プロンプトの生成ルール＋few-shot例で指示 |
| バリデーション | 40文字超えを warning 化 |

### ヒューリスティクス閾値

| ルール | 条件 | 閾値 |
|---|---|---|
| enthusiastic_short | 乗り気なのに平均文字数が短すぎる | **12文字**未満 |
| reluctant_long | 面倒がっているのに平均文字数が長すぎる | **30文字**超過 |
| indifferent_long | 興味薄なのに平均文字数が長すぎる | **30文字**超過 |
| indifferent_question | 興味薄なのに質問している | 質問が**1つ以上** |
| over_length | 一発話が上限を超えている | **40文字**超過 |

## 4.6 会話バリデーション

GPT出力を**ルールベース検証**（構造的正当性）と**ヒューリスティクス検証**（品質チェック）の二層で検査し、違反があればリトライ用フィードバックを生成する。**最大1回リトライ**。

### ルールベース（error でリトライ発動）

| ルール | 検証内容 |
|---|---|
| initiator_first | 先頭発話者が主導者であること |
| turn_count | 発話数が12〜16（8未満/20超はerror） |
| first_person | 一人称の表記揺れ検出 |
| invalid_speaker | 発話者が参加者に含まれること |
| memory_summary | 記憶の要約が空でないこと |

### ヒューリスティクス（warning のみ）

| ルール | 条件 |
|---|---|
| heuristic_enthusiastic_short | 乗り気なのに平均文字数が12文字未満 |
| heuristic_reluctant_long | 面倒がっているのに平均文字数が30文字超過 |
| heuristic_indifferent_long | 興味薄なのに平均文字数が30文字超過 |
| heuristic_indifferent_question | 興味薄なのに質問が1つ以上 |
| heuristic_over_length | 一発話が40文字超過 |

### リトライの仕組み

- error が1つでもあれば `valid=false` でリトライ発動
- フィードバックをベースプロンプトの末尾に `【再生成指示】` セクションとして追加（recency bias活用）
- 2回目が失敗してもその出力をそのまま使用（無限リトライ防止）

**実装参考**: `packages/shared/logic/conversation-validator.ts`。

## 4.7 会話スレッド管理

同一ペアの会話の「文脈継続性」を追跡する。スレッドステータスは**コード側で完全に決定**し、GPTの判断には依存しない。

### スレッドステータス

| ステータス | 意味 | 次回の扱い |
|---|---|---|
| ongoing | 約束あり（次回履行） | `continuation` 話題候補として高スコアで再登場 |
| done | 完了 | 話題候補にならない |

> ※ 旧仕様の `paused` ステータスおよび GPT の `signals`（continue/close/park）出力は**廃止**。スレッド進行はコード側の約束フラグで決定する。

### 約束生成システム

会話の中で「具体的な約束・予定」が生まれるかどうかを、コード側の確率抽選で決定する。約束が生成された場合のみ、2話完結のアーク（約束→履行）が発生する。

**約束発生確率:**

```
P(約束) = 10% × 関係重み × 話題重み × favorModifier
```

**関係重み:**

| 関係 | 重み |
|---|---|
| なし / 認知 | 0 |
| 友達 | 1.0 |
| 家族 | 1.1 |
| 親友 / 恋人 | 1.3 |

**話題重み:**

| 話題ソース | 重み |
|---|---|
| shared_interest | 1.1 |
| personal_interest | 1.0 |
| snippet | 0.8 |
| 季節・時事 | 0.7 |
| self_experience | 0.5 |
| 世間話 | 0.4 |
| continuation / third_party / 自己開示・質問 | 0 |

**好感度補正:**

```
favorModifier = 0.5 + (favor / 100)
→ favor=0: 0.5、favor=50: 1.0、favor=100: 1.5
```

実効レンジ: **約2〜17%**

### 3種の会話タイプとスレッド制御

| 会話タイプ | 条件 | threadStatus | プロンプト指示 |
|---|---|---|---|
| 通常 | 約束フラグOFF | **done** | 具体的な約束・予定は作らない。その場で完結する会話にすること |
| 約束生成 | 約束フラグON | **ongoing** | 会話の中で自然な約束・予定を1つ含めること。unresolvedThreadsにその内容を記述 |
| 約束履行 | continuation選択時 | **done** | 前回の約束を遂行する/した内容にし、この会話で完結させること |

- 大多数の会話は1話完結（done）
- 約束発生時のみ2話完結のアーク（ongoing → done）
- **3話以上続くスレッドは原理的に発生しない**

**実装参考**: `apps/web/lib/drizzle/schema.ts`（`topicThreads` テーブル）。

## 4.8 会話記憶

各会話の結果を構造化データとして保存し、次回の会話プロンプトに注入する「自己構築型記憶」。

### データ構造

| フィールド | 型 | 説明 |
|---|---|---|
| summary | string | 会話の要約 |
| topicsCovered | string[] | 扱った話題のリスト |
| unresolvedThreads | string[] | 約束内容（約束フラグON時のみ使用。通常会話では空） |
| knowledgeGained | Array<{about, fact}> | 得られた知識（対象キャラID付き） |

### ライフサイクル

```
会話N: GPT出力 → meta.memory に構造化記憶を格納 → events に保存
    ↓
会話N+1: loadConversationHistory() → 前回の memory を取得
    → buildUserPrompt() → 【前回の会話記憶】セクションに注入
    → GPT が前回の記憶を踏まえた会話を生成
    → 新しい memory を出力 → 繰り返し
```

### 鮮度管理

- `topicsCovered` → 直近5件のイベントから `recentTopics` を構築
- 同じ話題の繰り返しを鮮度ペナルティ（-3）で自動回避

**実装参考**: `packages/shared/types/conversation-generation.ts`（`conversationMemorySchema`）。

## 4.9 LLMプロンプト設計

全ゲームシステムのデータを1つのテキストプロンプトに統合し、LLMに「キャラクターらしい自然な会話」を生成させる。

### プロンプト構造（8セクション）

| 位置 | セクション | バイアス |
|---|---|---|
| 1（先頭） | 【最重要: 口調ルール】 | primacy bias |
| 2 | 【話題と会話の方向】 | |
| 3 | 【登場人物（補足情報）】 | |
| 4 | 【会話設定】 | |
| 5 | 【直近の共有スニペット】（条件付き） | |
| 6 | 【前回の会話記憶】（条件付き） | |
| 7 | 【生成ルール（厳守）】 | recency bias |
| 8（末尾） | 【出力仕様】 | |

### 日付情報の追加（SectionBの決定事項）

会話設定に**日付情報を追加**する（現行は時間帯・天気のみで日付がない）。

### 主要な生成ルール

- **12〜16ターン**の会話を生成
- 1発話は20〜30文字（上限40文字）
- 句点・感嘆符・疑問符で文を区切る
- 会話の冒頭を毎回同じパターンにしない（多様な始め方）
- **会話は自然に収束させること**（話題が一段落ついたところで終わる）
- **具体的な約束・予定は作らないこと**（約束フラグONの場合を除く。§4.7参照）
- 口調ルールの語尾・頻出表現は、文の流れに沿って自然に反映する（語尾の機械的な付与はしない）
- 一人称は指定表記を厳守

### GPT呼び出し仕様

| 項目 | 値 |
|---|---|
| モデル | GPT-5.1 |
| temperature | 0.8 |
| 出力形式 | JSON Schema (`strict: true`) |
| リトライ | 最大1回（バリデーション失敗時） |

### サニタイズ

GPT出力の微妙なスキーマ逸脱に対するレジリエンス層:
- `threadId`/`participants` をコード側の値で上書き
- `topic` が空なら `"雑談"` にフォールバック
- `speaker` が participants 外なら正規化
- `meta.memory` が欠けていれば空オブジェクトを挿入

**実装参考**: `packages/shared/gpt/prompts/conversation-prompt.ts`、`packages/shared/gpt/conversation-output.ts`。
