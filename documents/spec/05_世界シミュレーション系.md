# 05. 世界シミュレーション系

---

## 5.1 睡眠スケジュールシステム

住人の睡眠サイクルを管理し、「現在 active / preparing / sleeping のどれか」を判定する。会話スケジューラーが「寝ている住人は会話候補から除外する」という基本ルールの根幹。

### 3つの状態

| 状態 | 意味 | 会話参加 |
|---|---|---|
| active | 活動中 | ○ |
| preparing | 就寝準備中 | × |
| sleeping | 睡眠中 | × |

### データ構造

```
SleepProfile = {
  baseBedtime:   "HH:mm"    // 基準就寝時刻
  baseWakeTime:  "HH:mm"    // 基準起床時刻
  prepMinutes:   0-180      // 就寝準備時間（分）、デフォルト30
  todaySchedule?: {
    date:     "YYYY-MM-DD"  // JST基準
    bedtime:  "HH:mm"       // 今日の実際の就寝時刻
    wakeTime: "HH:mm"       // 今日の実際の起床時刻
  }
}
```

### デフォルトプリセット

| 生活傾向 | 就寝 | 起床 | 準備時間 |
|---|---|---|---|
| morning（朝型） | 22:30 | 06:30 | 30分 |
| normal（通常） | 00:00 | 08:00 | 30分 |
| night（夜型） | 02:30 | 10:30 | 45分 |

### 毎日のランダム化

- `baseBedtime` / `baseWakeTime` から **±60分** の範囲でランダム生成
- 24時間の剰余で日跨ぎに対応
- **正午（12時）を日替わり基準**とする（深夜に日付が変わってもスケジュールが急変しない）

### 状態判定ロジック

1. `todaySchedule` が未設定/古い場合 → `'active'`（安全なデフォルト）
2. `prepStart = (bedtime - prepMinutes + 1440) % 1440`
3. `bedtime ≤ 現在 < wakeTime` → `'sleeping'`（日跨ぎ対応）
4. `prepStart ≤ 現在 < bedtime` → `'preparing'`
5. それ以外 → `'active'`

### JST固定設計

`Date.now() + 9h` でJSTに変換。サーバー・クライアント間のタイムゾーン差を吸収。

**実装参考**: `packages/shared/logic/schedule.ts`（`generateTodaySchedule()`, `calcSituation()`）。

## 5.2 天候システム

ゲーム世界の天気を自動遷移させ、天気変化時にランダムな住人が「一言コメント」を残す。

### 4種の天気

`sunny`（晴れ）/ `cloudy`（曇り）/ `rain`（雨）/ `storm`（嵐）

### マルコフ遷移テーブル

| 現在 \ 次 | sunny | cloudy | rain | storm |
|---|---|---|---|---|
| **sunny** | 40 | 35 | 20 | 5 |
| **cloudy** | 25 | 35 | 30 | 10 |
| **rain** | 10 | 30 | 40 | 20 |
| **storm** | 5 | 25 | 30 | 40 |

各行の重みを合計してルーレット選択。同じ天気に留まる確率が最も高い（慣性）。

### スケジューラー

- **1時間 ±10% ジッター**で自動遷移
- `localStorage` ロック（TTL 5分）+ 最低54分間隔
- 静穏時間帯（全住人の睡眠から算出）中は更新停止

### 天気コメント

- 天気変化時に起きている住人をランダム選出し、gpt-5-miniで一言コメントを生成
- 住人の性格・口調に合わせた独り言（1-2行）
- コメント生成が失敗した場合は天気変更自体も行わない
- コメント住人が後から寝た場合、次回tickで `'Zzz…'` に自動置換

### データ保存

`world_states` テーブル（固定ID: `'00000000-0000-4000-8000-000000000000'`）に天気状態・静穏時間帯・コメントをJSONBで保存。

**実装参考**: `apps/web/lib/scheduler/weather-scheduler.ts`、`packages/shared/logic/weather.ts`。

## 5.3 共有スニペット生成

住人ペアの「偶然の共有体験」を生成する。**LLMを使わず**、場所×状況×修飾語の組み合わせテンプレートから合成する。

### パーツ

- **場所**（15個）: 商店街, 駅前, コンビニ, 公園, カフェ, 図書館, スーパー, 川沿い, バス停, 本屋, パン屋, 花屋の前, 階段の踊り場, 自販機の前, ベンチ
- **状況**（15個）: 偶然すれ違った, 隣の席になった, 一緒に帰ることになった 等
- **修飾語**（10個）: たまたま, 久しぶりに, いつもの 等

理論上 15×15×10 = 2,250パターン（相性マトリックスで制限される）

### 相性マトリックス

各場所ごとに許容する状況のインデックスを定義。不自然な組み合わせ（「図書館で雨宿り」など）を排除。

### 生成確率（関係別）

| 関係 | 確率 |
|---|---|
| none | 0%（生成しない） |
| acquaintance | 10% |
| friend | 30% |
| best_friend | 50% |
| lover | 60% |
| family | 50% |

### 鮮度チェック

同一ペアの前回生成から**6時間未満**はスキップ。

### 出力形式

`"{modifier}{place}で{situation}"` — 例: 「たまたまカフェで隣の席になった」

### ソース推定

| キーワード | ソース |
|---|---|
| 雨宿り / 信号 / 電車 | environment |
| 偶然 / 同時 / 同じ | coincidence |
| その他 | routine |

**実装参考**: `packages/shared/logic/snippet-generator.ts`、`shared_snippets` テーブル。

## 5.4 最近の出来事（直近イベント生成）

全住人の「最近あったこと」を定期的にLLMで生成し、知識伝播システム経由で他の住人に伝える。会話中の「〇〇さん、最近△△らしいよ」のような間接情報を生むための素材。

### 生成仕様

| 項目 | 値 |
|---|---|
| LLMモデル | gpt-5-mini（`temperature: 0.7`） |
| 鮮度チェック | **12時間**未満ならスキップ |
| 生成件数 | 0〜1件/人（全員に作る必要はない） |
| 文字数 | 20文字以内のゴシップ風事実 |
| 制約 | 「大事件は避け、日常の些事」「第三者が噂で伝えられる内容」 |
| 効率 | 全キャラ分を1回のLLM呼び出しでバッチ生成 |

### データ構造

```
recent_events テーブル:
  id, character_id, fact, generated_at,
  shared_with (JSON: 伝播済み受け手のID配列),
  updated_at, deleted
```

### 処理フロー

1. 鮮度チェック（12h）
2. 全住人データ + presetsをロード
3. gpt-5-mini で一括生成
4. `recent_events` に保存
5. `propagateKnowledgeBatch()` で知識伝播
6. 伝播結果を `offscreen_knowledge` に保存

**実装参考**: `apps/web/lib/batch/generate-recent-events.ts`。

## 5.5 知識伝播システム

住人Cに起きた出来事を、CとAの関係性・Cの性格に基づいて「Aが知る」かどうかを確率判定する。画面外（オフスクリーン）で住人同士が情報をやりとりしている世界観を表現する。

### 伝播確率の計算式

```
P(伝播) = intimacy × 0.5 + sociabilityFactor × 0.3 + expressivenessFactor × 0.2
```

| 因子 | 計算方法 | 範囲 |
|---|---|---|
| intimacy | `INTIMACY_WEIGHT[関係性]` | 0.0〜0.8 |
| sociabilityFactor | `(情報源のsociability - 1) / 4` | 0.0〜1.0 |
| expressivenessFactor | `(情報源のexpressiveness - 1) / 4` | 0.0〜1.0 |

### 具体例

| ケース | 確率 |
|---|---|
| lover + sociability=5 + expressiveness=5 | **90%** |
| acquaintance + sociability=1 + expressiveness=1 | **7.5%** |
| none（関係なし） | **0%**（即時リターン） |

### 設計上の特徴

- **情報源の性格に依存**: 受け手ではなく**情報源**の社交性・表現力が決定要素（「お喋りで表現豊かな人の出来事は広まりやすい」）
- **二重伝播防止**: `shared_with` 配列で既に伝播済みの受け手をチェック
- **関係性の対称性**: A→Bの関係があればB→Aも同じtypeで利用

### 伝播された知識の保存

```
offscreen_knowledge テーブル:
  id, learned_by(UUID), about(UUID),
  fact(text), source('offscreen'),
  learned_at, updated_at, deleted
```

会話プロンプトに「知っていること」として注入される。

**実装参考**: `packages/shared/logic/knowledge-propagation.ts`。

## 5.6 旧システムの廃止

**`experienceEvents` / `residentExperiences` テーブルは削除対象**。

設計思想として「事実だけ軽く生成して、キャラの解釈はGPTが性格情報を見て判断する」方針が採用された。旧システムの `hookIntent` に相当する機能は、既存のスタンス決定の仕組みで対応可能。`self_experience` の話題化は `recent_events` から直接引っ張る形で実装する。

## 5.7 システム間のフロー

```
[1時間ごと] 天候スケジューラー
              ├─ 全住人 sleepProfile → 静穏時間帯算出
              ├─ マルコフ遷移 → 天気変化
              └─ gpt-5-mini → 天気コメント → world_states保存

[毎日/アクセス時] 睡眠スケジュール
              ├─ ±60分ランダム抽選
              └─ active / preparing / sleeping 判定

[会話生成トリガー時] バッチ生成（インライン）
              ├─ generateSnippetsIfStale (6h)
              │    → 場所×状況×修飾語 → shared_snippets
              └─ generateRecentEventsIfStale (12h)
                   ├─ gpt-5-mini → recent_events
                   └─ propagateKnowledgeBatch
                        → offscreen_knowledge → 会話プロンプトに注入
```
